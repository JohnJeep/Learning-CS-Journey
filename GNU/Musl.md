## musl 是什么

**musl** 是一个轻量级、快速、简洁、注重标准兼容性的 **C 标准库**（libc）实现，用来替代 glibc（GNU C Library）或 uClibc。

C 标准库是 Linux 系统中最底层的基础库之一——任何用 C/C++ 写的程序（包括 shell、BusyBox、内核工具链本身）在运行时都要链接某个 libc，来提供 `malloc`、`printf`、`open`、`pthread` 之类的系统调用封装和标准函数。musl 就是 glibc 的一个"轻量替代品"。


### musl vs glibc 的核心区别

| 维度          | glibc                                         | musl                                                 |
| ------------- | --------------------------------------------- | ---------------------------------------------------- |
| 体积          | 较大（几 MB）                                 | 小得多（几百 KB）                                    |
| 静态链接      | 官方不太推荐，体积膨胀、有 NSS 相关问题       | 天生为静态链接优化，生成的二进制体积小               |
| 标准兼容性    | 功能全面，但历史包袱多，行为有很多"扩展/怪癖" | 严格遵循 POSIX/C99 标准，行为更可预测                |
| 性能/内存占用 | 功能全但运行时开销相对大                      | 更省内存，适合资源受限设备                           |
| 生态兼容性    | 事实标准，几乎所有软件都测试过                | 部分依赖 glibc 特有扩展的软件可能需要打补丁才能编译  |
| 典型使用场景  | 桌面/服务器 Linux 发行版（Ubuntu、CentOS 等） | 嵌入式系统、容器镜像（Alpine Linux）、静态二进制分发 |


### 为什么嵌入式/容器场景偏爱 musl

1. **体积小** —— 嵌入式设备 flash/存储空间有限，musl 编译出来的 rootfs 能小很多
2. **静态链接友好** —— 可以把程序和 libc 打包成一个完全静态的二进制文件，没有运行时动态库依赖问题；
3. **代码简洁，安全审计容易** —— 代码量远小于 glibc，历史遗留代码少，被认为更容易审计
4. **Alpine Linux 就是用 musl** —— 这也是 Alpine 容器镜像做得特别小（几 MB）的原因之一
